# Golang
## Slice
### 切片扩容策略
切片扩容遵循以下原则：

+ 如果期望容量大于当前容量的两倍，则直接使用期望容量；
+ 当前切片的长度小于1024，扩容为原来的两倍；
+ 当前切片的长度大于1024，扩容为原来的1.25倍；

### 切片拷贝
如果将一个较长长度的切片拷贝到较短的切片中，**不会发生扩容**，只会拷贝目标切片长度的数据；

### 数组与切片的区别
数组：

1. 数组是不可变长，在声明时需要指定数组的长度。
2. 数组是值类型，将一个数组赋值给另一个数组时，发生的是值拷贝。修改一个数组不会影响另一个的值。
3. 数组的元素拥有连续的内存空间。

切片：

1. 切片是对数组的引用。可变长度。
2. 将切片赋值给另一个变量时，也是值传递，但是他们引用的是相同的底层数组，因此对数据进行修改会影响另一个的底层数据。
3. 切片的底层数据在需要时会进行动态扩展。

## Map
### 扩容的条件
+ 负载因子大于6.5（增量扩容）
+ 哈希桶的数量（2^B）小于2^15时，溢出桶的数量大于2^B（等量扩容）
+ 哈希桶的数量（2^B）大于2^15时，溢出桶的数量大于2^15（等量扩容）

### 扩容过程
**渐进式扩容**，不会一次性搬迁全部数据，而是在每次访问数据时，顺便搬迁**一两个桶的数据，**将庞大的扩容成本分摊到多次操作中。

**增量扩容**：创建一个新的buckets数组，容量为原来的两倍，将旧数据逐步迁移到新桶中

**等量扩容**：buckets数量不变，重新做一次**数据搬迁**操作，将松散的键值对重新排列。

### 查找元素的过程
1. 首先计算键（key）的哈希值。
2. 随后取其低8位确定存储桶的位置。如果正常桶中没有找到，则会去溢出桶寻找。
3. 找到存储桶后取高8位与桶中tophash进行比较（tophash是为了加速查找），如果找到，则获取索引，找到索引处的key，对比结果。随后获取索引处的值。

:::warning
关于map中的key是**可以比较**的，因为不同的键可能产生相同的哈希值，也就是哈希冲突，此刻找到对应的桶后还需进行key之间的比较获取指定的值。

:::

### map的扩容时机
1. 装载因子（键值对数量/桶的数量）大于6.5，触发**双倍扩容（增量扩容）**。
2. 溢出桶的数量过多：
    1. 2^B<2^15，溢出桶的数量大于2^B，触发**等量扩容**；
    2. 2^B<2^15，溢出桶的数量大于2^15，触发**等量扩容**；

## Select（多路复用）
case语句中随机挑选一个未阻塞的通道执行相应的操作。**已经完成读取并且关闭的通道执行读操作也会被判定为非阻塞，这类读操作被视为“可立即执行”**。

```go
func main() {
    select {
    }
}
```

当协程陷入永久阻塞状态后，go语言有**死锁检测机制**，当其判定该协程**没有机会被唤醒**后，会引发panic。

### channel在什么情况下会引起内存泄漏？
channel引起的内存泄露是由于引起goroutine泄露间接引起的内存泄露。当一个goroutine一直阻塞在channel上，会出现内存泄漏。

### 什么时候关闭channel会引起panic？
1. 关闭一个已经关闭的channel。
2. **关闭一个只能接收的channel**。
3. **关闭一个nil值的channel**。

## Mutex
**互斥锁**用于**并发程序**对**共享资源**进行**访问控制**。

### 除了Mutex，还有哪些方式安全读写共享变量？
**信号量、channel、原子操作**

+ 信号量实现和Mutex差不多，主要通过信号量计数来保证；
+ channel是go语言最为推崇的方式，通过通信来数据所有权，更适合复杂的业务逻辑；
+ 原子操作针对最简单的整型或指针进行无锁操作，**性能最高**，常用于实现**计数器**或**状态位。**

### 原子操作是如何实现的？
**依赖底层CPU提供的原子指令**

sync/atomic包中的函数在**编译**时会被编译成硬件的**单条原子机器指令**。对应 LOCK；ADD这样的指令，直接锁住总线或缓存行，其他CPU核心不能访问这块内存，保证整个操作的原子性。

### sync.Once()作用是什么？
确保函数在整个程序的生命周期中只运行一次。

```go
type Once struct {
    done uint32  // 标识位
    m    Mutex
}
```

Once.Do(f)首次被调用时：

+ 首先通过原子操作快速判断标志位是否为1，为1说明执行过不再执行；
+ 为0则加锁，再次判断标志位，如果为0则执行函数，随后通过原子操作将标志位done置为1；
+ 最后释放锁

### waitgroup怎样实现协程等待？
本质上是一个计数器和信号量。计数器归零释放信号量唤醒协程。

## Go面向对象怎么实现？
通过**结构体**和**接口**实现面向对象的特性。

1. Go语言通过接口来定义对象的行为，而不是通过继承关系。一个类型只要实现了该接口的方法，就被定义为实现了该接口。

3. 通过字段首字母的大小写控制包外的访问权限。

## Go并发
+ 进程：操作系统**资源分配**的**最小单位**。拥有独立的内存空间，不同进程数据**不能直接共享，**需要通过管道或者消息队列共享数据。
+ 线程：操作系统**调度**的最小执行单位。同一个进程中的线程共享内存，可以通过内存地址直接共享数据。
+ 协程：用户态线程。由**Go调度器进行管理，**在Go语言中，使用的是CSP并发编程模型，通过通信来共享内存，而不是通过共享内存来进行通信。通过通道来实现通信。

### 进程与线程的区别
+ 进程是资源分配的最小单位，线程是调度的最小执行单位。
+ 进程的创建，切换，销毁 要比线程慢得多。
+ 进程拥有独立的资源，而线程没有资源，线程可以访问进程的资源。

### 协程与线程的区别
+ 协程的初始栈大小只有2KB，而线程默认的初始栈大小为1MB。
+ 线程切换开销大，涉及用户态与核心态的切换，而协程切换仅需保存用户上下文，完全在用户态完成。
+ 线程和进程都是同步机制，而协程是异步机制。
+ 协程不被操作系统内核管理。完全是程序控制。

### Go语言的并发模型
1. 轻量级协程goroutine，由**go运行时管理。**创建、切换、销毁开销更小。
2. channel实现通信。遵循CSP并发编程模型。**提供安全、同步的方式在goroutine之间传递数据**。
3. 多路复用：select语句实现在多个通道中选择一个执行。**有效处理多个协程的并发操作，避免阻塞**。
4. 互斥锁：提供sync包，用于在多个goroutine中**互斥**访问共享资源。
5. 原子操作：sync包下的atomic包，包括一系列的原子操作，用于在不加锁的情况下进行安全的并发操作。

### Goroutine过多会怎么样？应该如何处理？
虽然goroutine是轻量级线程，但是过多仍会出现**内存耗尽（即使空闲也会占用栈空间等资源）、调度开销大（GMP调度模型需要在多个Goroutine之间进行切换）**、**资源竞争与阻塞放大（例如：同时竞争同一把锁）**。

**控制goroutine数量的方式**：

1. 使用带缓冲的channel控制并发数。

使用缓冲通道模拟信号量，开启协程需要获取信号量，没有信号量则进入阻塞状态，等待其他协程执行完毕释放信号量才能继续执行。

2. 使用工作池复用goroutine。

预先创建一定数量的goroutine，通过channel接收任务并进行处理，避免频繁的创建和销毁协程。

```go
package main

import (
	"fmt"
	"sync"
)

func main() {
	const (
		workerCount = 50   // 固定 Worker 数量
		taskCount   = 1000 // 任务总数
	)

	tasks := make(chan int, taskCount) // 任务队列
	var wg sync.WaitGroup

	// 启动 Worker
	for i := 0; i < workerCount; i++ {
		wg.Add(1)
		go func(workerID int) {
			defer wg.Done()
			for task := range tasks { // 从任务队列取任务
				fmt.Printf("Worker %d 处理任务 %d\n", workerID, task)
			}
		}(i)
	}

	// 发送任务到队列
	for i := 0; i < taskCount; i++ {
		tasks <- i
	}
	close(tasks) // 任务发送完毕，关闭队列

	wg.Wait()
}
```

## init()函数什么时候执行？
**简答**：在main函数之前

**详细**：runtime初始化每个导入的包，初始化不是按照导入顺序，而是**按照解析的依赖关系**。**<font style="color:#DF2A3F;">没有依赖的包最先初始化</font>**。

每个包首先初始化包作用域的**常量和变量**，然后执行包的init()函数。同一个包，或者同一个源文件有**多个init函数**。init函数**没有参数和返回值**，不能被其他函数调用，**同一个包内的init函数调用顺序不能保证**。

执行顺序：import——>const——>var——>init()——>main

一个文件可以有多个init()函数

## Go语言普通指针和unsafe.Pointer有什么区别？
普通指针*int *string有明确的类型信息，**编译器**会进行**类型检查**和**垃圾回收追踪**，不同类型的指针之间不能直接转换，**较为安全。**

unsafe.Pointer是**通用指针类型**，unsafe.Pointer可以与任意类型的指针进行转换。

**通用指针收到GC管理和类型的约束，unsafe.Pointer不受类型的约束但是受GC追踪。**

:::warning
总结就是unsafe.Pointer与普通指针相比不受类型的约束，可以随意转换。但是会被GC跟踪。

:::

## unsafe.pointer与uintptr有什么区别和联系
可以相互转换，这是GO提供的唯一合法的指针运算。典型的用法是，先将unsafe.Pointer转换为uintptr做算数运算，然后在转化为unsafe.Pointer使用。

:::warning
**最关键的区别是GC跟踪。**unsafe.Pointer会被GC跟踪，指向的内存不会被**错误回收**；而uintptrGC不知道他指向的是什么，如果没有其他引用，可能会被**错误回收**；

:::

**unsafe.Pointer有GC保护，uintptr没有。**

## go语言是如何通过channel和goroutine实现高效并发的？
Go语言遵循CSP并发编程模型：通过通信共享内存，而不是通过共享内存进行通信。

Goroutine：goroutine非常轻量，初始栈大小仅为2KB，由Go运行时管理，创建、切换和销毁成本开销极低。其次，它具有高效的调度模型GMP，在GMP调度模型之前没有P（处理器），M运行G需要从全局队列中获取。会产生严重的并发冲突，因此需要为全局队列加锁，致使整个存取变成串行化。P拥有本地队列，M与P绑定，运行P中的G，这时就解决了全局队列的并发冲突问题。此外，还有work stealing机制确保资源利用率。

Channel：无缓冲通道在发送和接收时都会被阻塞，实现天然的同步。channel的发送和接收都是原子的，无需额外加锁。此外，select多路复用能够同时监听多个channel，实现非阻塞通信。

## Interface
### 类型转换和类型断言的区别
类型转换需要转换双方的类型具有兼容性，是在**编译期**的**强制转换**，而类型断言是在**运行期**提取接口的类型。

**安全性差别很大**：类型转换在**编译期**保证安全性，而类型断言可能在**运行时**失败，需要对返回值做判断。

**使用场景不同**：类型转换主要用于字符、字符串、整型之间的转换问题；类型断言主要将interface{}转换为具体类型进行相关操作。

**底层实现不同**：类型转换是对**数据的重新解析**，而类型断言是**检查接口的类型信息**。

## GMP
Go**运行时的核心调度模型**。

**调度逻辑**：M必须与P绑定才能执行G，G维护一个本地队列P，M轮转执行P里面的G，当本地队列中的G执行完后，M会从全局队列或者其他P的本地队列中偷取一半G继续执行（work stealing机制）。

### P和M在什么时候会被创建？
P：在**调度器初始化**时一次性创建，会根据**gomaxprocs**创建指定数量的P。

M：采用**按需创建**的策略，当P需要M绑定运行G时，并且线程池中没有空闲的M供其使用，就会创建新的M。

### M0
m0是Go启动时创建的第一个线程，他在Go程序的整个生命周期都存在。

m0负责Go程序的**<font style="color:#DF2A3F;">启动流程</font>****，**包括**调度器的初始化，垃圾回收设置，内存管理器初始化，**他会创建并运行第一个goroutine执行main.main函数。在**程序执行期间，**会参与goroutine调度，和其他M没有本质区别。m0在程序退出时还**负责清理工作**，比如等待goroutine结束，执行defer函数。

### G0
g0和普通的协程不同，他是**调度协程。**每个M都有自己的g0，使用的是线程的栈空间，大约8KB。

**核心作用**：专门负责**调度逻辑**，包括goroutine的创建、销毁、调度等，**当M需要调度时**，会从用户协程切换到g0执行**schedule（）函数**（负责调度协程的函数）。同时G0还负责处理垃圾回收，栈扫描、信号处理等运行时操作。

## 内存管理
内存逃逸是指一个变量在函数中创建，在函数外引用。其**生命周期**超过了函数范围，因此该变量的内存应该分配在堆上而不是栈上。

### 内存逃逸
编译器在编译期间根据**逃逸分析策略**，将原本分配到栈上的对象分配到堆上。

主要逃逸场景：

+ 返回局部变量的指针
+ 闭包引用外部变量
+ 切片map动态扩容
+ 大对象：超过栈大小限制的对象直接分配在堆上。
+ interface类型。

### 内存逃逸有什么影响？
**堆对象需要垃圾回收释放内存，栈对象会跟随函数结束被编译器回收，****<font style="color:#DF2A3F;">大量的内存逃逸会给gc造成压力</font>。**

### channel分配在栈上还是堆上？
分配在堆上，**channel被设计用来进行协程间的通信**，**其生命周期和作用域不会局限于单个函数中**，一般都分配在堆上。

### 什么是闭包？
指一个函数能够**捕获并访问外部作用域定义的变量**，即使该外部函数返回（也就是执行完毕），这些变量仍然可以**被内部函数引用和修改**。

### 什么情况下发生内存泄漏？
1. **goroutine泄漏**：goroutine没有正常退出，可能是由于channel一直陷入阻塞状态导致goroutine一直没有成功退出。
2. **channel阻塞**：channel一直陷入阻塞状态，导致goroutine一直阻塞，内存无法回收。
3. **slice引用大数组**：slice引用一个大数组中的一小部分，导致整个数组的内存无法被GC回收。解决方法是使用copy创建一个新的切片。
4. **map元素过多**：map中删除元素只是标记为删除，如果一个map元素很多后来元素减少，会占用大量内存。
5. **定时器未停止**：time.After或time.NewTimer创建的定时器如果不手动停止，会在堆中持续存在。
6. **循环引用**：Go的GC能够处理循环引用，但是在一些特定情况下仍然会出现问题。

### 内存可见性与CPU缓存机制
**问题**：定义一个全局变量，开启两个协程对其进行读取和修改，协程A对其进行修改，协程B对其进行读取，读取到的还可能是初始值。

**原因**：现代CPU都是多核的，每个核心都有自己的**高速缓存**。

+ 当协程A在CPU核心1上修改变量时，修改的值首先被写入核心1的缓存中，**可能不会立即同步到主内存**。
+ 协程B在CPU核心2读取变量，从核心2的缓存或主内存中读取，**核心1的修改没有同步，读取到的就是旧值**。

## 垃圾回收
### 常见的垃圾回收方式
所有GC算法可以归结于**追踪GC**和**引用计数**两种。

+ 标记清除：从根对象出发，存活的对象做标记，清扫可以回收的对象。
+ 引用计数：根据引用对象自身的计数进行回收，当引用计数归零进行回收。

### 三色标记法
三色定义：

+ 白色：未被访问的对象，垃圾回收结束后白色对象会被回收。
+ 灰色：已经被访问的对象，但是其引用对象还未被访问，是待处理的队列。
+ 黑色：已被访问并且其引用对象已被访问，垃圾回收结束后保留的对象。

GC流程：GC开始时，将所有对象标记为白色，随后从根对象（全局变量，栈上的对象）开始访问，将其标记为灰色，访问其引用的对象。访问后将其引用对象标记为灰色，添加到灰色队列中，根对象标记为黑色。随后一直按照这个方法执行，知道对象只存在黑色和白色，最后回收白色对象。

### GC的根对象是什么？
全局变量、执行栈（每个协程都有自己的执行栈，执行栈上包含栈上的变量及指向分配的堆内存的指针）、寄存器。

### 如何解决并发标记清除时，用户并发修改对象引用问题的？
+ 强三色不变性：不存在被黑色对象引用的白色对象
+ 弱三色不变性：存在被黑色对象引用的白色对象，但是需要灰色对象引用白色对象，对白色对象进行保护。

**写屏障和弱三色不变性（Go1.8之后采用混合写屏障规则：插入写屏障和删除写屏障）**

1. GC开始时将栈上的对象全部标记为黑色。
2. GC期间，栈上新建的对象标记为黑色。
3. **被添加的对象**标记为灰色。
4. **被删除的对象**标记为灰色。

### Go语言中的GC如何调优？
1. **复用已经申请的内存**，使用sync.pool复用需要经常重复创建的对象。
2. 调整GOGC，适量将GOGC的值设置的更大，让GC触发的事件变得更晚，**减少其触发频率**，增加用户代码对机器的使用频率。

# MySQL
## 主从复制
主要用于读写分离，数据备份，架构扩展。

**主从复制主要涉及三个线程**：**binary log dump**、**IO thread**、**SQL thread**

**主从复制的流程**：从节点发送同步请求（**<font style="color:#DF2A3F;">start slave</font>**），请求中同时包含上一次同步的文件名以及偏移量，主节点创建**binary log dump**线程，用于将指定文件名和指定偏移量后的数据发送给从节点，返回的数据中还包括这次同步到的文件名以及偏移量。从节点的**IO thread**负责接收数据保存到本地的**relay log**日志文件中，同时将文件名和偏移量保存到本地的**master-info**中。随后**SQL thread**检测到relay log中发生更新，便会解析执行relay log文件中的内容，实现数据同步。

## reids如何实现高可用
1. 主从复制，**主从复制流程**：

主库与从库断联期间，会将收到的写操作命令写入repl_backlog_buffer中。从库在请求同步时，会将当前的slave_repl_offset发送给主库，主库会判断master_repl_offset与slave_repl_offset之间的差值，如果差值大于repl_backlog_buffer，会进行全量复制，否则进行增量复制。

    1. 全量复制：主库从库进行第一次复制，从库与主库建立连接，随后发送请求同步（**<font style="color:#DF2A3F;">sync</font>**）命令。主库执行bgsave命令生成rdb文件，期间收到的写操作命令会写入内存缓冲区中。随后将rdb文件发送给从节点，从节点首先清除自身的旧数据，再加载rdb文件到内存中。最后将缓冲区的写操作命令发送给从数据库，从库执行写操作命令完成同步。
    2. 增量复制：从节点重新连接后，发送上一次同步的偏移量，主库会将master_repl_offset和slave_repl_offset之间的数据发送给从节点。
    3. 命令传播：主从节点进入稳定期之后的同步方式采用命令传播。主**节点每执行一次写操作，都会将操作命令同步给从节点**。
2. 哨兵模式：**一组独立的进程**用来监控主从节点的状态，当发现主节点宕机，会自动选取一个从节点晋升为新的主节点，通知其他从节点修改相关配置，**实现故障的自动转移**。一旦主节点恢复，根据需要将其配置为从节点，提供故障恢复和负载均衡。
3. 切片集群：**当数据量大到一台服务器无法承载**，可以采用切片集群的方案，将数据分布在不同的服务节点上。具体流程：将数据集分割为16384个槽，每个节点负责一部分槽的数据。在进行键值对的存取时，首先计算键的CRC16哈希值，随后确认对应的槽，随后客户端直接与槽所对应的节点进行通信，执行相关的操作命令。切片集群采用主从复制机制，一个槽有一个主节点和多个从节点，向主节点写入数据，会被**异步**的复制到从节点，确保数据的安全性以及高可用。此外，切片集群使用哨兵模式检测节点健康状态，当节点不可用时，会重新分配槽到可用节点。

## 数据库和缓存一致性
cache aside策略：分为读策略和写策略

读策略：读取数据时首先查询缓存，命中直接返回结果，没有命中查询数据库。

写策略：更新数据库中的数据，删除缓存中的数据。

**为什么更新数据库后不更新缓存**？存在并发事务导致数据不一致性：事务A首先更新数据库，事务B此时到来更新数据库和缓存，此时事务A再更新缓存，破坏了数据库和缓存的一致性。

4. **<font style="color:#DF2A3F;">高并发场景下，并发冲突严重</font>****。**

分布式锁是在分布式场景下**协调多个节点对共享资源的访问控制**。

**死锁的解决方案：**为获得锁的操作设置一个超时时间，超过这个时间自动释放锁。

**看门狗机制**：用于**检测和恢复系统的异常状态**。对于分布式系统而言，他会检测获取锁的线程或进程是否在正常工作，如果很长时间没有响应或释放锁，看门狗会强制释放该线程持有的锁。

为每个客户端设置唯一ID防止误解锁。

3. 程序改写。
4. 引入Nosql和程序架构调整。

**硬优化：**

1. 提升系统硬件：磁盘IO，网络带宽，CPU

**分库分表：**

1. 先根据业务情况而定，只分表是否能够满足业务需求，但是分表只能解决单张表过大的问题，并不能解决单机数据库的磁盘IO以及网络带宽受限问题。如果不能够满足需求，再进行分库。

结论：在数据量及访问压力不是特别大的情况下，先进行参数优化，查询语句优化，程序优化，如果满足不了需求，在考虑架构优化（架构优化成本较高）

+ **建立索引**
+ **避免使用select ***
+ **选择合适的数据类型**
    - 尽量使用tinyint，smallint，mediumint代替int类型。
    - varchar按需分配。
    - 尽量使用整数或者枚举代替字符串类型。
    - 尽量少量使用null，很难查询优化并且额外占用索引空间。

## 如何防止升级为表锁？
### 行锁为什么会升级为表锁？
行锁是粒度最细的锁，但是数据库为了**减少锁管理的开销**，满足一定条件时，会将大量锁合并为表锁。

1. **锁定行数过多**：单个事务锁定的行数超过阈值。
2. **无索引，索引失效**：行锁依赖索引，如果查询条件没有走索引，数据库会做全表扫描。
3. **锁等待超时，死锁**：大量行锁等待时，数据库可能触发“快速失败”机制，升级为表锁；
4. **特殊SQL场景**：update、delete不加where条件，where条件使用函数或表达式导致索引失效。批量操作一次性锁定大量行。

### 解决方法
1. 确保操作走索引
2. 控制单词锁定的行数（避免单词锁定大量的行）核心是分批次，小批量操作
3. 缩短锁的持有时间，

# Redis
## 混合持久化
混合持久化中，当执行**bgrewriteaof**命令或者**aof**文件大小超过阈值时，会进行aof重写。首先执行bgsave命令开启一个后台线程生成rdb文件，在重写期间，redis会将接收到的写操作命令写入重写缓冲区中，将rdb数据写入AOF文件的上半部分,随后将缓冲区的写操作命令以追加的形式写入AOF下半部分。当不在aof重写期间，数据库接收到的写操作命令会首先进入AOF缓冲区，随后刷入aof文件中。

## AOF的写回策略
No：执行写操作命令，将写操作命令写入AOF日志缓冲区，由操作系统决定写回时机。

EverySec：执行写操作之后，将写操作命令写入AOF日志缓冲区，每隔一秒写回磁盘。

Always：同步写回，在执行写操作命令后，同步将AOF日志数据写回磁盘。

## redis主从复制
主从复制模式主要有，全量复制，增量复制，命令传播。**在首次执行同步时，会进行增量复制**。其次，在主库与从库断联期间，主库会将接收到的写操作命令写入repl_backlog_buffer中，在进行主从复制时，从数据库会向主库发送自己的slave_repl_offset，主库会计算它与master_repl_offset，当差值大于repl_backlog_buffer时，进行全量复制，当差值小于repl_backlog_buffer时,进行增量复制

1. **全量复制**：从库向主库发送psync请求同步命令，同时发送自己的slave_repl_offset偏移量，主库判断是进行全量复制，随后执行bgsave命令生成rdb文件，发送给从库，从库清空自己的内存数据，将rdb文件加载到内存实现同步。
2. **增量复制**：主库判断是进行增量复制，会将偏移量后面的数据发送给从库。

## 哨兵模式
哨兵模式是一组独立的进程用于检测主从节点的状态，当发现主节点发生故障时，选取一个从节点晋升为新的主节点。同时更改其他从节点的相关配置，更改为新的主节点。实现故障的自动转移。当旧的主节点恢复时，根据需要将其作为从节点加入集群中，提供**故障恢复**和**负载均衡**。

## 切片集群
切片集群中是将整个数据集分为16384个槽，将槽分布到不同的节点上。对键值对进行修改时，首先计算键的CRC16哈希值，找到对应的槽，客户端会直接与负责该槽的节点进行通信，执行相关操作，不需要中间代理。槽采用主从复制的模式，有一个主节点和多个从节点，实现切片集群的高可用性。

## 集群脑裂
主节点与从节点断连，但是主节点与客户端之间的连接是正常的，客户端会继续与主节点进行通信，此时主节点将数据写入缓冲区。当哨兵发现主节点断联，会选取一个新的主节点，随后主节点恢复连接，哨兵会将他降级为从节点，从节点与主节点进行数据同步操作，第一次同步采用全量复制，会清空内存中的数据，导致缓冲区中的数据丢失，这就是集群脑裂。

## 如何防止集群脑裂
设置主节点与从节点的**最小连接数**以及**延迟阈值**，当小于最小连接数或者超过延迟阈值时，不再接收客户端的请求，等到新的主节点上线，由他与客户端建立连接继续通信。

## 内存淘汰策略
1. 不进行内存淘汰：内存不足时，返回错误。**默认的淘汰策略**。
2. 进行数据淘汰：
    1. VolatileTTL：优先淘汰更早过期的键值
    2. VolatileLRU：对带有过期时间的键使用LRU淘汰策略，其他键使用Noeviction（最近最少使用）
    3. VolatileLFU：对带有过期时间的键使用LFU淘汰策略，其他键使用noeviction（使用频率最低使用淘汰策略）
    4. VolatileRandom：对带有过期时间的键使用随机淘汰。
    5. AllkeysLRU：对所有键使用LRU淘汰算法
    6. AllkeysLFU：对所有的键使用LFU淘汰算法
    7. AllkeysRandom：对所有的键使用随机淘汰策略；

## Redis事务
一组有序执行的redis命令。执行期间不会被其他客户端的命令打断。主要使用muti和exec执行。

+ multi：开启一个事务。
+ redis命令：这些命令不会立即执行，而是放入一个队列中，等待执行。
+ exec：开始执行一系列命令，这些命令作为一个事务一起执行。如果发生错误，全部取消。
+ 事务执行成功，会返回一个所有命令执行结果的数组。
+ 执行失败，回滚到执行开始前。

在事务中，可以使用watch监控一个或多个键，在执行期间其他客户端对该键进行修改，事务会被取消。确保事务执行的一致性。

## 重传机制
### 超时重传
在指定时间内未收到ack报文段，重新发送数据。

### 快速重传
不以时间为驱动，而是以数据为驱动，当接收到三次相同的ack报文段后，重新发送数据。但是面临一个问题：**重传之前的一个，还是所有的**？

### SACK选择性确认
将已接受到的消息发送给发送端，这样发送端就知道哪些数据丢失。

### DSACK
使用SACK告诉发送方哪些数据被重复接收了。

## TCP滑动窗口
用于接收方向发送方表示自己还有多少缓冲区可以接收数据。表示自己的数据处理能力。

### TCP流量控制
通过滑动窗口判断接收方的接收能力，动态调整发送速率。

## 拥塞控制
网络出现拥堵时，继续发送大量数据包，会导致数据包**延迟丢失**等，TCP重传数据，重传之后加剧网络阻塞。**防止发送方数据填满整个网络**。

## 粘包和拆包
TCP是面向字节流协议，底层不会维护**消息边界**，仅按字节序列传输。粘包拆包的核心原因：

+ TCP为了提高发送效率，当有多个小数据包时，会进行缓冲，到达一定大小再进行发送，这就是粘包。
+ 拆包，当一个数据包过大时，超过传输上限，会进行拆分。

解决方法：在应用层定义消息边界。

+ 固定发送的长度
+ 定义数据的格式为“头”+“体”，头用来存储数据包的长度，按照长度读取即可。
+ 使用特殊的分隔符进行区分。	

# Docker
Docker是一个开源的容器化平台，核心作用是**将应用程序极其依赖打包成标准化的容器，**使得应用可以在任何支持docker的环境中**一致运行**。

+ 镜像：容器的模板，包含应用程序的代码，配置，依赖等。
+ 容器：镜像的运行实例。
+ 仓库：存储和分发镜像。

# 微服务与Web服务的区别？
Web服务特点：核心是跨语言，跨系统的交互式服务。

1. 功能聚合。

微服务特点：微服务是一种**架构设计模式**，核心是将单体应用拆分为**独立部署、自治的小型服务**。

1. 服务拆分：按照业务逻辑进行拆分，每个服务专注于单一职责（**高内聚**）服务间通过轻量协议通信。
2. 独立部署：每个服务可以单独开发、测试、部署，不依赖于其他服务。
3. 技术栈灵活：不同的服务可以选择适合自己的技术栈，不受统一框架约束。
4. 弹性设计：**强调容错性（如熔断、降级）**、**可扩展性**（单服务扩容）。
5. 数据自治：每个服务都有自己独立的数据库（数据隔断）。
